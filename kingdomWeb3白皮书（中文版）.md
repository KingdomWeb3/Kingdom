## 摘要

Kingdom 是一个结合去中心化社交网络与边缘 AI 推理算力市场的 Web3 平台。用户通过日常观看和互动行为，在完全无感的情况下贡献设备闲置算力；平台聚合这些算力，为 B 端客户提供低成本边缘推理服务。创作者通过观看者自动支付的代币获得收益，代币价值锚定真实算力需求，形成可持续的经济闭环。

Kingdom 不依赖广告，不贩卖用户数据，代币 KING 通过 B 端付费回购销毁实现通缩。项目致力于构建一个全民可参与、创作者可盈利、算力可高效利用的去中心化社交计算网络。

## 1. 背景与问题

### 1.1 社交网络的困境

现有主流社交平台（如 TikTok、YouTube）采用广告驱动模式，平台抽取绝大部分收入，创作者分成比例极低，用户贡献注意力和数据却无任何回报。Web3 社交项目（如 Lens、Farcaster）虽然将数据主权归还用户，但其代币经济模型缺乏真实价值锚定，代币价格主要依赖投机预期，难以形成可持续发展的生态。

### 1.2 AI 推理算力市场的结构性矛盾

随着大模型应用进入规模化落地阶段，AI 推理需求呈爆发式增长。据行业报告预测，2030 年边缘 AI 市场规模将达到 2700 亿美元。然而，算力供给与需求之间存在严重错配：

- **高端算力稀缺**：大型 AI 公司抢占 H100 等旗舰 GPU，中小团队无力承担高昂的云推理费用（如 AWS Rekognition）。
- **低端算力闲置**：全球数十亿台消费级设备（手机、PC）绝大多数时间处于空闲状态，其 NPU、GPU 算力未被有效利用。

### 1.3 现有去中心化算力网络的局限

已有的 DePIN 算力项目（如 Render Network、Akash）解决了算力供需匹配问题，但缺乏用户粘性和天然流量入口；而 SocialFi 项目具备用户活跃度，却无可持续的变现模式。将二者结合，形成“社交引流 + 算力变现”的双轮驱动，是 Kingdom 解决上述问题的核心思路。

## 2. 解决方案

### 2.1 双闭环经济模型

Kingdom 设计了两条相互增强的经济闭环：

**社交闭环（C2C）**：
- 用户质押少量 KING 代币获得高级账户（王国公民），可发布内容。
- 观看内容时，系统自动从观看者账户扣除微量 KING 转账给创作者（用户无感，通过 Session Key 实现）。
- 创作者获得 KING，可继续持有或兑换法币。

**计算闭环（B2C）**：
- B 端客户（AI 公司、开发者）用 USDC 发布边缘推理任务。
- 用户设备在闲置时自动接收并执行微任务，提交 ZK 证明或乐观验证结果。
- 验证通过后，设备获得 KING 奖励（30% 即时释放，70% 锁仓 3 个月）。
- 平台收到 USDC 后，70% 用于公开市场回购 KING 并销毁，30% 注入社区金库。

### 2.2 代币价值锚定

KING 代币的价值直接锚定边缘推理任务的市场价格（例如 1000 次图片识别定价 1 KING）。B 端客户必须购买 KING 才能支付任务费用，持续产生买压；平台用 USDC 回购销毁 KING，形成通缩机制。代币持有者的利益与网络算力需求高度一致。

### 2.3 协议手续费

所有链上经济活动（观看支付、推理任务、代币兑换、NFT 附魔等）抽取 0.2%～1% 手续费，分配比例为：
- 社区金库（用于回购和生态建设）：30%
- 团队激励池：22.5%
- 算力奖励池：27.5%
- 创作者基金：20%

## 3. 技术架构

### 3.1 总体设计

KingDOM 采用三层分离架构：

| 层级 | 职责 | 技术实现 |
|------|------|----------|
| 链上结算层 | 身份管理、内容存证、推理任务市场、代币发行与回购 | Polygon / Arbitrum 智能合约 |
| 客户端（阅读器） | 用户交互、本地推理执行、内容渲染、无感支付 | React Native / Flutter + ONNX Runtime + Session Keys |
| 存储与分发层 | 内容存储、模型分发、视频缓存 | IPFS + Filecoin + P2P 网络 |

### 3.2 关键技术创新

#### 3.2.1 去中心化边缘推理验证

- 混合验证机制：高价值任务（医疗影像等）使用 ZK 证明（ezkl、RISC Zero）；海量低价值任务（图片分类等）采用乐观验证 + 随机抽查。
- 异构算力调度：自动将任务匹配到手机 NPU、PC GPU 等不同设备。

#### 3.2.2 零 Gas 用户体验

- 基于 ERC-4337 账户抽象，用户无需持有主链代币。
- Paymaster 代付 Gas 费用。
- Session Keys 实现观看支付无需每次签名。

#### 3.2.3 可扩展内容类型（Content Header）

- 每个内容携带 2-4 字母内容头（如 `VD` 视频、`PT` 文本帖子），阅读器根据头字段动态加载渲染组件。
- 社区可通过 DAO 添加新的内容头，阅读器开发者可自由支持。

#### 3.2.4 观看即支付：内容解码与代币结算的原子化绑定

传统 Web3 内容平台将内容存储与支付分离，观看行为无需链上结算，创作者收入依赖用户主动打赏或平台补贴。KingDOM 引入了一种新的内容消费范式：**观看行为与代币支付原子化绑定**。

**流程设计**：
1. **内容发布**：创作者在客户端完成视频转码、切片、生成 HLS 播放列表。原始分片存储于 IPFS/Arweave，播放列表 CID 及内容元数据（含内容头）上链存证。
2. **观看触发**：用户点击播放时，阅读器调用链上 `ContentRegistry` 的 `payForView` 接口，从用户账户向创作者账户转账微量 KING（例如单次观看 0.001 美元等值）。
3. **原子化执行**：支付交易被打包后，事件触发阅读器的解码模块。阅读器根据 CID 从存储网络拉取内容分片，完成解密（若有）和解码，开始播放。
4. **无感体验**：用户通过 Session Keys 预先授权小额支付额度，无需每次签名；Paymaster 代付 Gas 费，用户完全无感知。

**技术要点**：
- **支付与解码的解耦**：为防止链上确认延迟影响播放体验，采用“乐观播放 + 信用质押”模式。用户授权信用额度（例如 100 KING），阅读器在链下先播放，后批量提交支付交易；恶意违约将扣除信用质押。
- **强制代币消耗**：所有观看行为（除新用户免费额度及每日免费前 N 次外）均需支付，为 KING 代币创造刚性、高频的应用场景。
- **创作者即时收益**：每笔观看支付的代币直接进入创作者钱包，无需平台抽成或延迟结算。

**相较现有方案的改进**：

| 方案 | 支付方式 | 用户体验 | 创作者收益稳定性 |
|------|----------|----------|------------------|
| Lens / Farcaster | 可选打赏 | 需主动操作 | 低（依赖粉丝意愿） |
| 传统流媒体订阅 | 月费 | 无感知，但内容受限 | 中等（平台抽成高） |
| **KingDOM 观看即支付** | **强制微支付（自动）** | **无感知** | **高（观看即有收入）** |

该设计使 KING 代币具备真实、高频、不可回避的消费场景，与算力挖矿形成闭环：用户贡献算力获得 KING → 观看内容消耗 KING → 创作者获得 KING → 创作者再将 KING 用于发布内容或兑换法币。代币在生态内持续流转，价值锚定日益增强。

## 4. 代币经济模型

| 参数 | 数值 |
|------|------|
| 代币符号 | KING |
| 总量 | 10,000,000,000 (100 亿) |
| 分配 | 社区金库 30%、团队 15%（2 年线性锁仓）、早期融资 7.5%、创作者基金 20%、算力奖励池27.5 % |
| 回购销毁 | B 端 USDC 收入的 70% 用于公开市场回购 KING 并销毁 |
| 协议手续费 | 0.2%～1%（具体视经济活动类型） |
| 手续费分配 | 社区金库 50%、团队激励池 30%、算力奖励池 10%、创作者基金 10% |

### 4.1 锁仓与释放

- **团队**：2 年线性解锁，第一次解锁发生在 TGE 后 12 个月。
- **种子轮投资者**：12 个月锁仓，之后 12 个月线性释放。
- **公募**：无锁仓或极短锁仓（视合规要求）。

### 4.2 价值捕获总结

- B 端客户购买 KING 支付推理任务 → 买入压力
- 平台回购销毁 KING → 通缩压力
- 观看支付消耗 KING → 流通速率下降
- 质押 KING 获得高级账户 → 锁定流动性
